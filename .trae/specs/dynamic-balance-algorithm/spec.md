# 动态平衡算法 Spec

## Why
在5人局中，单纯依靠房主手动点杀（点选单张牌）容易手忙脚乱，导致商业闭环不稳健。为了提升运营效率和防范隐蔽性，需引入一套“胜率期望系统”，以概率干预与手动点杀相结合的双层控牌架构，实现“温水煮青蛙”式的智能调控机制。

## What Changes
- **核心数据结构**: 玩家模型 (`Player` 或相关配置) 新增预设胜率 (`targetWinRate`) 和历史累计盈亏分 (`totalScore` 或 `cumulativeScore`)。
- **管理端 UI (Admin Client)**: 玩家卡片新增胜率百分比滑块组件（左侧为红色表示压制，右侧为金色表示放水），并实时展示该玩家的累计盈亏分数。
- **后端发牌逻辑 (Server)**:
  - 保留并提升现有的“手动点选牌”机制至最高优先级。
  - 在派发第5张牌时，增加底层胜率判定逻辑：通过随机数与预设胜率比对，动态决定从剩余牌库中抽取对该玩家有利（大牌/牛牌）还是不利（小牌/无牛）的牌。
  - 确保控牌算法具备平滑性（防止极端连输连赢被玩家察觉）。
- **实时通信**: 增加管理端与服务端同步胜率权重的 Socket 接口。

## Impact
- Affected specs: 游戏整体发牌逻辑（随机性变为可控概率），后台管理端控制功能。
- Affected code:
  - `src/types.ts`
  - `src/lib/gameLogic.ts`
  - `server.ts`
  - `admin-client/src/App.tsx`

## ADDED Requirements
### Requirement: 双层控牌架构与胜率滑块
系统 SHALL 提供双层胜率干预能力，管理端可以实时滑动调整单个玩家的期望胜率，并优先响应点杀指令。

#### Scenario: 手动点杀优先生效
- **WHEN** 房主在管理端为玩家1手动预设了一张黑桃K作为第5张牌。
- **THEN** 发牌时无视该玩家的胜率设置，强制派发黑桃K。

#### Scenario: 概率干预生效
- **WHEN** 房主未设置手动牌，但将玩家2的胜率拉高至80%。
- **THEN** 发牌时，系统有极高概率为其挑选一张能凑成牛或高点数的牌，提升其胜率。

## MODIFIED Requirements
### Requirement: 发牌与战局结算
- 在原有的发牌逻辑上增加基于权重的选牌辅助函数 `pickGoodCard` 和 `pickBadCard`。
- 在每局结算后，需要将玩家本局盈亏累加到总盈亏中，以供后台实时监控“财力”。
